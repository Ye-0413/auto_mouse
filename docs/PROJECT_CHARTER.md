## 1. 立项决议

在现有仓库基座上，正式立项建设 **Windows 优先的跨应用轻量 RPA 客户端**：Excel 批量数据驱动、流程 DSL、本地 SQLite 持久化、可观测执行记录；**不**在一期内承诺云端调度、多租户、拖拽式低代码画布、全自动 AI 修流程（见 `docs/BACKLOG.md`）。

**一期目标**：达成 `docs/DEVELOPMENT_PLAN.md` 第 3 节「MVP 成功标准」中可在 **受控 Windows 环境** 手工验收的端到端闭环；开发机可为 macOS，但 **发布与验收以 Windows 为准**。

## 3. 一期里程碑（建议顺序）

与 `DEVELOPMENT_PLAN.md` 第 4 节阶段编号对齐，**立项后优先补齐缺口**：

1. **M1 — 执行与可靠性**：失败截图开关落地、行级状态与日志字段与 MVP 标准对齐；Windows 上窗口激活/输入路径实测（`pygetwindow` 等依赖写进安装说明）。
2. **M2 — 条件引擎（P6）**：`if` / 安全比较 DSL，无裸 `eval`；与 Runner 整合；单测覆盖。
3. **M3 — 浏览器策略（P7）**：Playwright CDP 或等价连接；步骤类型与 `ActionResult` 统一；fixture 页回归。
4. **M4 — 批量产品化（P8）**：勾选行执行、暂停/取消、结果导出、与「最小业务闭环」场景对齐（见 `project.md` 相关章节）。
5. **M5 — 录制 MVP（P9）**：最小事件采集 → 生成可编辑步骤 → 存 Flow（可降级为坐标/热键）。
6. **M6 — 打包与交付（P10）**：PyInstaller、`README`/`ACCEPTANCE`、干净 Windows 安装跑通。

里程碑允许**并行**：M2（条件）与 M3（浏览器）在 Step schema 稳定后可分人并行；录制 M5 可在 M4 首批交付后插入。

## 4. 立项后首批可开工任务（PR 粒度）

以下顺序减少返工，便于冷启动代理直接领单：

| 顺序 | 任务 | 依赖 | 验收要点 |
|------|------|------|----------|
| A | 失败截图：配置项 → Runner/桌面层 → `executions` 路径与日志 | 当前 Runner | 失败行有可追溯截图路径或明确无权限原因 |
| B | 条件引擎初版：步骤类型 `if`、轻量表达式、单元测试 | Flow schema 协商 | 无 `eval`；文档中列举支持的运算符 |
| C | Playwright 最小适配：CDP 连接 + `open_url` 后读文本/点击 | 依赖注入与 `run_step` 分支 | fixture HTML 自动化测试一条 |

完成 A→B→C 后，再开 **导出与暂停（M4）** 或 **录制 MVP（M5）** 的独立 PR。

---

## 5. 风险与决策

| 风险 | 缓解 |
|------|------|
| macOS 开发与 Windows 行为差异 | CI 或定期在 Windows 实机跑 `pytest` + 一条 E2E 清单 |
| Excel 大表内存与耗时 | 已：整表在后台线程加载；续：可选分批或进度条 |
| Step schema 来回改 | 变更需同步 `flow_dsl`、`step_dialog`、Runner、迁移说明 |

**策略冻结点**：条件语法与 Playwright 步骤的 JSON 形状在 **M2/M3 合并前** 评审一次，记入 `docs/DEVELOPMENT_PLAN.md` 版本历史。
